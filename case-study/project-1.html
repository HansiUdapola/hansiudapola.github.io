COMNAP — Transforming a frozen Unity viewer into a live, interactive 3D web platform

The original system trapped users in a loading screen before showing anything useful. I rebuilt it from the ground up — replacing a Unity WebGL monolith with a React + Three.js platform backed by Firestore — and turned a passive viewer into an interactive, admin-owned data tool.

01. The Problem

The COMNAP system existed to visualise station and research data in a navigable 3D environment. The technical choice was Unity WebGL — reasonable at the time, but it came with a fundamental UX flaw baked into the architecture: users saw nothing until the entire application had downloaded.

In UX, a blank screen is a trust problem. Users have no way to know whether something is happening, how long it will take, or whether the page is broken. For a system that was supposed to communicate data clearly, the entry experience communicated nothing.

Three compounding problems

Problem 01 - Load Experience
The "Wait" Problem

Unity WebGL required the entire application bundle to download before rendering a single pixel. There was no loading indicator, no progress state, and no partial content — just a blank page. In UX terms, a few seconds of an empty screen is enough to lose a user permanently.

Problem 02 — Interactivity
Static Data, Static Thinking

The viewer was exactly that — a viewer. Users could look at 3D station data but couldn't filter it, search within it, or interact with individual data points in any meaningful way. A map had become the endpoint rather than the beginning of an interaction.

Problem 03 — Admin Dependency
The Developer Bottleneck

Any update to the data displayed in the system — adding a station, correcting coordinates, updating attributes — required a developer to manually edit source files and redeploy. Admins had zero autonomy. Every data change became a technical ticket.

02. Research & Discovery

Before writing a line of code, I needed to understand what the system had to do — not just what it currently did. I mapped the full user journey across two audience types: end users navigating the 3D environment, and admins responsible for maintaining the data behind it.

Technical audit

I audited the Unity WebGL build to understand the exact source of the load problem. The bundle included the full Unity runtime, all 3D assets, and application logic in a single download — with no streaming, no chunking, and no partial rendering path. The architecture made progressive loading structurally impossible without a complete rebuild.

Admin workflow mapping

I ran structured walkthroughs with the admin team to document their current process for updating station data. The baseline: 4–6 steps involving a developer, a code editor, a deployment pipeline, and a manual data file — to add a single station. The dependency was total.

Competitive and technology review

I surveyed how comparable spatial data platforms handle progressive loading — particularly how Google Maps, Mapbox, and Cesium implement tile-based and LOD (level-of-detail) loading strategies. The pattern was consistent: low-fidelity representation first, high-fidelity progressively.

03. Key Insights

"The loading screen is the product, as far as most users are concerned. If that experience is broken, nothing else matters."

— UX principle driving the rebuild strategy
  • Architecture was the UX problem. The loading issue wasn't a design symptom — it was baked into the Unity WebGL architecture. No amount of UI polish could fix a monolithic download. The system needed to be rebuilt, not reskinned.
  • Perceived performance matters more than actual performance. A 3-second wait with visible progress feels faster than a 2-second blank screen. The goal wasn't just to load faster — it was to give users something real to interact with immediately, regardless of total load time.
  • Admin autonomy unlocks product velocity. Every developer-dependent data update was a bottleneck on both sides — admins waited for changes, developers were interrupted by non-engineering work. A no-code data layer would free both.
  • The 3D space should be a database, not a picture. The system's value was in the data behind the geometry, not the geometry itself. Making that data filterable, searchable, and queryable was the real opportunity.

04. AI As an Engineering Efficiency Multiplier

The scope of this rebuild — porting 3D rendering logic, rebuilding a data backend, and designing a new admin interface — was significant for a single UX engineer. I used AI deliberately and strategically to compress timeline without sacrificing quality, focusing my own attention on the decisions AI couldn't make: UX judgement, interaction design, and user-facing polish.

Use 01
Math & Logic Translation

AI translated Unity C# camera and coordinate transformation logic into Three.js equivalents — a task that would have taken weeks of manual porting. I validated and adjusted the output; AI compressed the mechanical work.

Use 02
Performance Logic

Used LLMs to design efficient async loading patterns — specifically the background GLTF swap logic — ensuring high-quality assets loaded off the main thread without blocking UI interaction.

Use 03
Admin Flow Stress-Testing

Before writing any CSS, I used AI to identify dead ends and ambiguities in the proposed "Add Station" admin flow. This surfaced 3 edge cases that would have caused user errors — fixed at the design stage, not the QA stage.

The principle: AI handled the mechanical and logical heavy lifting; I handled every decision that touched a human being. The result was a quality bar that would have required a larger team to reach on a traditional timeline.

05. Design Process

Stack selection

The new stack — React + Vite + Three.js + Google Firestore — was chosen to solve each identified problem at the architectural level, not through workarounds on top of a broken foundation.

  • React + Vite: Component-based UI for the data layer, search, filters, and admin dashboard. Vite's build optimisation gave fast development iteration and a lean production bundle.
  • Three.js: Native browser 3D rendering with full control over loading strategy — no Unity runtime overhead, no monolithic download requirement.
  • Google Firestore: Real-time NoSQL backend that let admins push data changes that appear in the 3D space immediately — no redeploy required.

UX improvement 1 — Progressive 3D loading

The signature UX improvement was a two-step loading strategy that made the application feel instant — even before the high-quality assets finished loading.

Step 01 — Instant
Low-fidelity GLTF + progress bar

A lightweight, low-poly version of the 3D map loads in milliseconds. Users see the environment immediately and can begin exploring. A progress bar shows the high-quality assets loading in the background.

↳ User is engaged within seconds
Step 02 — Background swap
Seamless high-quality swap

While the user explores the low-poly map, high-fidelity GLTF assets load asynchronously on a separate thread. When ready, they swap in seamlessly — no visible transition, no interruption to the user's session.

↳ No loading anxiety, no blank screen

UX improvement 2 — No-code admin dashboard

The admin dashboard was designed for a non-technical user — someone who needed to add, edit, or remove station data without understanding the underlying 3D coordinate system or data schema.

The key design decisions: a guided form with inline validation that mapped human-readable inputs (station name, location, category) to the 3D coordinate system automatically; a preview of where the station would appear in the 3D space before saving; and empty state guidance for first-time admins that walked through the process without requiring a manual.

UX improvement 3 — From map to searchable database

A React UI layer over the Three.js canvas added the interaction model the original system lacked entirely: real-time search, category filters, and clickable data points that surfaced detailed station information in a side panel — without leaving the 3D view.

06. Solution

The rebuilt COMNAP platform replaced every element of the original system without changing what the product fundamentally did — it still put 3D station data in front of users. But the experience on every dimension was transformed.

  • Progressive two-step GLTF loading — low-poly instant render followed by seamless high-quality background swap
  • React UI layer with real-time search, category filtering, and interactive data points over a Three.js canvas
  • No-code admin dashboard connected to Firestore — station data changes appear in the 3D space in real time, no developer involvement required
  • Guided "Add Station" flow with coordinate abstraction, inline validation, and 3D preview before save
  • Contextual tooltips and empty state guidance across admin features — no manual needed
  • Fully responsive layout — the 3D environment and admin tools work correctly on all screen sizes
  • WCAG-conscious UI layer — keyboard-navigable search, readable contrast ratios, screen-reader-labelled 3D controls

What this project demonstrated

COMNAP was an exercise in full-stack UX engineering — where the design decisions and the engineering decisions were inseparable. The progressive loading strategy required both a UX insight (perceived performance > actual performance) and an architectural choice (GLTF streaming + async thread management). The admin dashboard required both a UX design (zero-to-one for non-technical users) and a data layer design (Firestore schema that mapped cleanly to the 3D coordinate system).

This is the kind of problem that sits in the gap between design and engineering — and where UX Engineering, as a discipline, earns its place.
